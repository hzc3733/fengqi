## 1. autonomous-ai-agents（自主 AI 编程 Agent）

### claude-code

**简介**：将编程任务委托给 Claude Code CLI（Anthropic 官方推出的自主编程 Agent）。Claude Code 可以读取文件、写代码、执行 shell 命令、管理 git 工作流，完全自主运行。

**核心概念**：
- **PTY 模式**：pseudo-terminal，伪终端。Claude Code 是交互式终端应用，必须配合 PTY 才能正常运行，否则会挂起。
- **工作目录隔离**：每个任务建议在独立目录运行，避免冲突。
- **后台模式**：使用 `background=true` 启动长时间任务，通过 `process` 工具监控进度。

**常用命令**：
```bash
# 单次任务（one-shot）
terminal(command="claude --message '实现登录功能'", workdir="~/project", pty=true)

# 后台运行
terminal(command="claude --message '重构认证模块'", workdir="~/project", background=true, pty=true)
```

**适用场景**：构建功能、代码重构、PR 审查、批量修复 issue。

### codex

**简介**：将编程任务委托给 OpenAI Codex CLI（OpenAI 官方推出的自主编程 Agent）。与 claude-code 功能类似，区别在于使用的模型和生态不同。

**核心概念**：
- **必须处于 git 仓库中**：Codex 拒绝在非 git 目录运行。
- **`--full-auto`**：沙箱模式，自动批准工作区内的文件变更。
- **`--yolo`**：无沙箱、无审批，最快也最危险。

**常用命令**：
```bash
# 单次任务
terminal(command="codex exec '添加深色模式开关'", workdir="~/project", pty=true)

# 并行修复多个 issue（使用 git worktree）
terminal(command="git worktree add -b fix/issue-78 /tmp/issue-78 main", workdir="~/project")
terminal(command="codex --yolo exec 'Fix issue #78: 描述'", workdir="/tmp/issue-78", background=true, pty=true)
```

### hermes-agent

**简介**：配置、扩展或贡献 Hermes Agent 本身的技能。这是唯一一个在询问"如何配置 Hermes"时必须加载的技能。

**核心概念**：
- **Gateway**：Hermes 的消息网关，负责协调各个平台（Telegram、Discord、CLI 等）的通信。
- **config.yaml**：主配置文件，控制平台启用、认证、工具集等。
- **skills**：可扩展的知识模块，存放在 `~/.hermes/skills/` 目录。
- **多租户（multi-agent）**：一个 Hermes 实例可同时服务多个用户或对话。

**常用命令**：
```bash
# 配置网关
hermes gateway setup

# 查看/设置配置项
hermes config get platforms.webhook.enabled
hermes config set platforms.webhook.enabled true

# 管理技能
hermes skills list
hermes skills enable <skill-name>
```

### opencode

**简介**：将编程任务委托给 OpenCode CLI（provider-agnostic 的开源 AI 编程 Agent，支持 OpenRouter、Anthropic、OpenAI 等多提供商）。

**核心概念**：
- **provider-agnostic**：不绑定特定模型提供商，可用任意 provider 的模型。
- **TUI/CLI 双模式**：`opencode` 启动交互式 TUI；`opencode run '...'` 为单次命令。
- **Session 恢复**：退出后可恢复历史会话。

**常用命令**：
```bash
# 单次任务
terminal(command="opencode run '实现支付功能'", workdir="~/project")

# 交互式后台会话
terminal(command="opencode", workdir="~/project", background=true, pty=true)
# 通过 process 工具发送指令
process(action="submit", session_id="<id>", data="实现支付功能")
```

## 2. creative（创意与设计）

### architecture-diagram

**简介**：生成深色主题的技术架构图，以独立 HTML 文件输出（内嵌 SVG 矢量图形），无需任何外部依赖或 API。

**核心概念**：
- **SVG**：Scalable Vector Graphics，矢量图形格式，可在浏览器中无损缩放。
- **组件语义配色**：前端用青色、后端用翠绿、数据库用紫色、云服务用琥珀色……便于一眼识别架构层次。
- **双矩形遮罩技术**：在半透明填充的组件上绘制箭头时，为防止箭头穿透，使用"不透明背景矩形 + 半透明样式矩形"两层实现。

**输出**：单个 `.html` 文件，包含内联 CSS 和 SVG，可直接用浏览器打开。

**适用**：软件系统架构图、云基础设施（VPC/子网/区域）、微服务拓扑、数据库 API 地图。

### design-md

**简介**：编写和验证 Google 的 DESIGN.md 文件——一种将视觉身份规范文档化的开放规范（Apache-2.0）。

**核心概念**：
- **Design Token**：设计标记（颜色值、字号、间距等），以 YAML 格式存储。
- **DTCG**：Design Tokens Community Group，W3C 标准化组织，制定了设计 token 的格式规范。
- **WCAG**：Web Content Accessibility Guidelines，网页内容无障碍指南，lint 时会检查对比度合规性。
- **Token 引用**：`{colors.primary}` 引用其他 token，避免硬编码值重复。

**文件结构**：YAML front matter（机器可读 token）+ Markdown 正文（人类可读的解释）。

**CLI 工具**：
```bash
npx @google/design.md lint DESIGN.md          # 验证结构+WCAG
npx @google/design.md diff DESIGN.md v2.md    # 对比差异
npx @google/design.md export --format tailwind > tailwind.theme.json  # 导出
```

### pixel-art

**简介**：将任意图片转换为复古像素画，并可选地制作成带环境特效（雨、雪、萤火虫）的循环动画视频。

**10+ 场景动画**：night（星空+萤火虫）、urban（雨+霓虹）、snow（飘雪）、underwater（气泡）等。

## 3. data-science（数据科学）

### jupyter-live-kernel

**简介**：通过 live Jupyter kernel 实现有状态的 Python REPL，变量在多次执行间持久化，适合数据探索、ML 实验、迭代式分析。

**核心概念**：
- **有状态 REPL**：不同于一次性脚本，每次 `execute` 后变量都保留，可逐步构建复杂分析。
- **hamelnb**：一个管理 Jupyter notebook 和 kernel 的工具，支持通过 CLI 操作 notebook（增删单元格、执行代码、查看变量）。
- **kernel**：Jupyter 的 Python 进程，负责执行代码并返回结果。

**关键命令**：
```bash
# 查看服务器和 notebook
uv run "$SCRIPT" servers --compact
uv run "$SCRIPT" notebooks --compact

# 执行代码（有状态）
uv run "$SCRIPT" execute --path scratch.ipynb --code 'import pandas as pd' --compact

# 查看变量
uv run "$SCRIPT" variables --path scratch.ipynb list --compact
```

**使用场景**：数据清洗、API 探索、ML 实验、迭代调试。

### webhook-subscriptions

**简介**：创建 webhook 订阅，让外部服务（GitHub、GitLab、Stripe、CI/CD 等）通过 POST 请求触发 Hermes Agent 执行任务。

**核心概念**：
- **Webhook**：一种服务器间推送机制，外部服务 POST 事件到指定 URL，触发预定义动作。
- **HMAC-SHA256**：一种签名验证机制，确保请求来自合法来源，防止伪造。
- **事件驱动架构**：外部事件触发 Agent 运行，无需轮询。

**设置流程**：
1. 在 `config.yaml` 或 `.env` 中启用 webhook 平台
2. `hermes webhook subscribe <name>` 创建订阅
3. 配置外部服务 POST 到返回的 URL

**常用场景**：GitHub issue → Telegram 通知、Stripe 支付成功 → Agent 处理、CI/CD 构建完成 → Discord 播报。

## 5. email（邮件）

### himalaya

**简介**：通过 Himalaya CLI 在终端管理邮件，支持 IMAP（接收）/ SMTP（发送），无需图形界面。

**核心概念**：
- **IMAP**：Internet Message Access Protocol，邮件接收协议，多设备同步邮件状态。
- **SMTP**：Simple Mail Transfer Protocol，邮件发送协议。
- **MML**：Mail Markup Language，Himalaya 的邮件模板格式，用于撰写富文本邮件。
- **pass**：Linux 密码管理工具，Himalaya 通过它安全存储 IMAP/SMTP 密码。

**常用命令**：

### systematic-debugging

**简介**：四阶段根本原因调试法——理解问题后再修复，不猜测。

**核心原则**：修复症状而不理解原因 = 失败。

**四阶段**：

**Phase 1：根本原因调查**
- 仔细阅读错误信息
- 稳定复现问题
- 检查近期变更（git log/diff）
- 追踪数据流找到问题组件
- **完成检查清单**：错误信息→复现→变更历史→证据→隔离到组件→形成假设

**Phase 2：模式分析**
- 找到类似的正常工作的代码
- 对比差异
- 理解依赖和配置

**Phase 3：假设与测试**
- 明确陈述假设
- 最小化改动测试假设
- 一个变量一次

**Phase 4：实现**
- 写回归测试
- 修复根本原因
- 验证修复 + 全量测试通过
- **3 次修复失败 → 质疑架构，而非继续修复**

### test-driven-development

**简介**：TDD——红绿重构循环，先写失败测试，再写最小实现。测试通过才实现，无测试不写代码。

**核心原则**：
- **铁律**：无失败测试不写生产代码
- **RED**：写一个展示缺失行为的失败测试
- **GREEN**：写最小代码让测试通过（允许硬编码、作弊）
- **REFACTOR**：清理代码，保持测试绿色

**常见理性化借口**（须克服）：
- "太简单不需要测试" → 简单代码也会坏，测试只需 30 秒
- "之后补测试" → 通过的测试不能证明任何事
- "手动测试过" → 手动测试是临时的，无法回归
- "删除浪费" → 沉没成本谬误，保留不可信代码是技术债

### writing-plans

**简介**：编写可执行的实现计划——精确文件路径、完整代码示例、准确命令和期望输出。

**核心原则**：
- **Bite-sized**：每任务 2-5 分钟专注工作
- **精确路径**：给出 `src/models/user.py` 而非"模型文件"
- **完整代码**：可复制粘贴，而非"添加验证函数"
- **精确命令**：含期望输出的完整 shell 命令
- **DRY / YAGNI / TDD**：不重复、不画蛇添足、测试驱动

**计划结构**：
```
# 特性名 实现计划

目标：一句话描述
架构：2-3 句方法
技术栈：关键技术和库

## Task N：任务名称
**文件**：
- 创建：`path/to/file.py`
- 修改：`path/to/existing.py:45-67`

**Step 1：写失败测试**
[完整代码]
**Step 2：运行验证失败**
期望：FAIL — "function not defined"
**Step 3：写最小实现**
[完整代码]
**Step 4：运行验证通过**
**Step 5：提交**
```
